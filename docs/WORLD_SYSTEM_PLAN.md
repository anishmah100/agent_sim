# World System — Plan (June 2026)

The pass to disentangle map design from the engine and make worlds
swappable.

## Decisions captured

| Decision | Choice |
|---|---|
| World scope | Full self-contained bundle. `worlds/<name>/` owns map config + scenario Go code + art pack manifest + rules. |
| Rule language | Rich predicates + combinators (declarative). Built-in predicate library + AND/OR/NOT + scalar functions (density, distance, altitude) + weights. |
| Layout templates | ASCII grid + sprite key. Maximally visual; the editor renders the same grid. |
| UI ceiling | Light visual editing — drag towns to move, click to drop new structures, side panel for properties, save back to YAML, regen button. |
| Scenario binding | Go code physically relocates under `worlds/<name>/scenario/`. Engine imports it through a clean interface. |

## Target directory layout

```
engine/                                      ← engine runtime only; no world-specific code
  internal/
    world/           — generic tile/entity state + tick + snapshot
    wire/            — WS protocol
    core/            — generic systems (combat, money, inventory, …)
    scenario/        — INTERFACE definitions + helpers; no concrete scenarios
    genrules/        — rule engine: predicates, combinators, evaluator (NEW)
  cmd/
    engine/          — main; imports the active world's scenario package
    genworld/        — declarative world generator (NEW; replaces genworld_pretty)

worlds/
  eldoria/                                   ← the swappable bundle
    world.yaml         — top-level: name, scenario, dims, seed, generator passes
    biomes.yaml        — biome definitions + placement rules + density tables
    towns.yaml         — town/landmark list (this is what the UI edits)
    templates/         — structure templates as ASCII grids
      capital_crossroads.yaml
      city_default.yaml
      village_triangle.yaml
      village_row.yaml
      village_square.yaml
      village_crossroads.yaml
      village_tower.yaml
      farm_homestead.yaml
      …
    npcs.yaml          — NPC archetype pools + spawn config
    art.json           — art pack manifest (extends ../../art/manifests/sprites.json)
    tiles.json         — generated map output (regenerated by genworld; checked in)
    scenario/          ← Go code; was engine/internal/scenario/fantasy_town/
      scenario.go
      verbs/
        …

art/                                         ← shared sprite library (no change)
  manifests/sprites.json
  processed/…

scenarios/                                   ← gone; contents moved into worlds/*/scenario/
```

A new world = drop a new directory under `worlds/<name>/`. If it reuses an
existing scenario, it can just point `scenario: fantasy_town` at the
existing path. If it has new verb logic, it gets its own
`worlds/<name>/scenario/` Go package.

## Config schemas

### `world.yaml` — top of the bundle

```yaml
name:      Eldoria
display:   "Eldoria"
scenario:  "github.com/anishmah100/agent_sim/worlds/eldoria/scenario"
dims:      [1500, 1500]
seed:      42
art_pack:  "sprites.eldoria.json"           # under art/manifests/, extends default

# The generator runs these passes in order. Each pass is a built-in
# pipeline step the engine provides. New PASS TYPES need engine code
# (rare); STAGES referenced here are pure data.
pipeline:
  - { kind: noise,     name: elevation, seed: 42, octaves: 4, scale: 0.003 }
  - { kind: noise,     name: moisture,  seed: 43, octaves: 3, scale: 0.005 }
  - { kind: biomes,    from: biomes.yaml }
  - { kind: river,     start: [250, 0], end: [1000, 1000], width: 5 }
  - { kind: river,     start: [1050, 1050], end: [1450, 700], width: 5 }
  - { kind: ocean,     side: east,  shore_jitter: 50 }
  - { kind: roads,     hub: crossroads, link_neighbours: true }
  - { kind: towns,     from: towns.yaml }
  - { kind: scatter,   from: scatter.yaml }    # wilderness farms / outposts
  - { kind: vegetation, from: biomes.yaml }    # density tables per biome
  - { kind: entities,  from: npcs.yaml }
  - { kind: validate }                          # run all placement rules
```

    vegetation:

        veg:tree_pine:  6
        veg:tree_birch: 3

        veg:bush_round_dark: 1

    wilderness_archetypes: [woodcutter, woodcutter, cloaked_wanderer, drifter, child]

rules:

## The rule engine (`engine/internal/genrules/`)

Predicates and combinators are just Go types implementing a single
interface:

```go
type Predicate interface {
    Eval(ctx EvalCtx) float64    // 0 = reject, 1 = perfect; smooth values
                                  // allow weighted scoring.
}
```

Built-in predicates (lots of them, all simple):
- `RequireTerrain([…])`, `ForbiddenTerrain([…])`
- `RequireBiome([…])`
- `MinDistance(kind, n)`, `MaxDistance(kind, n)`
- `RequireNear(kind, { max_distance, weight })`
- `InRect(x0, y0, x1, y1)`
- `NoiseInRange(name, [min, max])`
- `Density(kind, [min, max])`
- `Or(…)`, `And(…)`, `Not(…)`

Adding a new predicate = one Go struct + one entry in a registry. The
YAML schema is decoded into a tree of Predicate values; the generator
calls `Eval` at every candidate placement.

This matches Minecraft's worldgen density-function pattern: declarative
config trees composed from a fixed library of primitive operations.

## Generator pipeline

```
┌───────────────────────────────────────────────────────────────────────┐
│  cmd/genworld -world worlds/eldoria/                                  │
└───────────────────────────────────────────────────────────────────────┘
           │
           ▼
   Read world.yaml ──┐
                     ▼
              ┌───────────────┐
              │ load biomes   │  noise maps, ocean, lake, rivers, biome layer
              │ load towns    │  hand-placed entries from towns.yaml
              │ load templates│  ASCII grids
              │ load npcs     │  pools + wilderness configs
              │ load rules    │  parse predicate trees
              └───────┬───────┘
                      ▼
         apply pipeline steps in order
           1. terrain noise → biome map
           2. carve features (rivers, ocean shoreline)
           3. carve road network (hub-and-spoke + neighbour links)
           4. stamp towns from towns.yaml using their templates
              - apply town's placement rules; nudge if violated
              - render ASCII grid relative to center
           5. scatter wilderness structures
              - sample candidate sites; apply rules; place if pass
           6. vegetation pass: biome density tables + weights
           7. NPC placement: town pools + biome wilderness pools
              + guarantee minimum per archetype
           8. validate: re-run all rules, log any violations
                      ▼
              write worlds/eldoria/tiles.json
              (plus a diff summary)
```

Determinism: every stochastic decision goes through the world's seed. A
regen with the same `world.yaml` + same `seed` produces identical output.

## The visual editor

```
http://127.0.0.1:5173/editor/eldoria
```

Reuses the existing chunked tilemap renderer at low zoom.

UI elements:
- Atlas canvas — drag to pan, scroll to zoom (same Pixi viewport).
- Top bar: world name, current pass dropdown (View / Edit Towns /
  Edit Templates), `Regen` button, `Save` button, undo/redo.
- Right panel: properties of the currently-selected entity (town,
  structure, etc.) as a form. Edit, hit Apply, redraw.
- Toolbar: `Add ▾` dropdown — pick a structure template, click on the
  map to drop it. Drag any existing structure to move it. Right-click
  to delete.
- Status bar: rule-violation warnings ("Pinewood new pos lands on water
  — nudged to (322, 462)").

Engine endpoints (new):
- `GET /api/v1/world/<name>/config` → bundled YAMLs as JSON
- `PUT /api/v1/world/<name>/config/towns` → write towns.yaml
- `PUT /api/v1/world/<name>/config/template/<id>` → write a template file
- `POST /api/v1/world/<name>/regen` → re-run generator, reload tiles.json

Client-side undo/redo stack keeps the YAML history; saves are atomic
file-writes through the engine endpoints.

## Migration plan for Eldoria

Goal: every visual change to the world goes through the new config. The
regenerated tiles.json should look IDENTICAL to current Eldoria (modulo
intentional clean-ups).

1. Create `worlds/eldoria/` directory.
2. Port `var towns []town` from genworld_pretty/main.go → `towns.yaml`.
3. Port each `stampTownCenter` switch arm → `templates/*.yaml` ASCII grids.
4. Port `biomeAt` y-bands → `biomes.yaml`.
5. Port `scatterFarms` → `scatter.yaml` + new pipeline step.
6. Port `addDeco` water/footprint checks → `rules:` in biomes.yaml.
7. Port the NPC pools → `npcs.yaml`.
8. Port engine/internal/scenario/fantasy_town/ → `worlds/eldoria/scenario/`.
9. Update `engine/cmd/engine/main.go` to import the active world's
   scenario by full Go import path. Update `./agent_sim start` to pass
   `-world worlds/eldoria/`.
10. Run `cmd/genworld -world worlds/eldoria/`; visual diff the result
    against the previous tiles.json. Fix discrepancies.
11. Boot the engine on the regenerated tiles.json; verify the 100-bot
    soak still holds 60 Hz.

## What this pass does NOT do

- **No declarative verb handlers.** Verbs stay Go code under the
  world's scenario package.
- **No runtime scenario hot-swap.** Compile-time selection only —
  active world is chosen at engine startup.
- **No second world implementation.** Eldoria is the only world we ship
  initially. Adding a second world after this lands should be a
  data-only change (drop directory, write YAMLs).
- **No multi-user collaborative editing.** Single editor instance,
  single user, local undo/redo.
- **No new sprite art.** Catalog stays as it is.
- **No engine API changes for agents/SDK.** This is a content
  refactor; the WS protocol is untouched.

## Phased execution

| Phase | What | Risk | Days |
|---|---|---|---|
| 1 | Data refactor — directory structure, port Eldoria into config files, move scenario Go code | medium (Go imports + many file moves) | ~1 |
| 2 | Rule engine package + new genworld tool — predicate trees, generator pipeline, regenerate Eldoria, visual diff | medium (new code, but pure Go) | ~1.5 |
| 3 | Visual editor — frontend route, drag/drop, side panel, save endpoints | high (UI work, new endpoints, atomic writes) | ~2 |
| 4 | Polish, docs, soak verification | low | ~0.5 |

Total: ~5 days of focused work.

The **lightweight** thing the user wanted is broken by the visual editor
phase. If we trim the UI to **view-only with structured info on click**
(no drag/drop, no save endpoints), phase 3 collapses to ~0.5 day and the
total comes in under 3 days. Worth a check before phase 3 starts.

## Open questions to revisit at phase boundaries

- **Validation strategy for editor moves.** When a user drags Pinewood
  into a lake, do we (a) hard-reject the save, (b) auto-nudge to the
  nearest legal tile and show a warning, (c) accept and let the rule
  validator complain on regen? I lean (b); confirm before phase 3.
- **Templates vs procedural fill.** Hand-arranged ASCII grids are
  precise but rigid. If we ever want "drop a hamlet here, generate its
  layout randomly", we'd add a `rules:` form alongside `grid:`. Out of
  scope for this pass; revisit when needed.
- **Live-reload of templates.** Should editing a template trigger a
  re-render of every town using it, or only on explicit regen? Probably
  explicit; auto could be expensive on a 1500×1500 world.
